# 現行機能・状態の棚卸し

## 現行の業務機能（確認済み）

| 領域 | 機能 | 新システムで残す業務価値 |
| --- | --- | --- |
| 案件 | 案件作成、所在地・期間・状態、案件ごとの情報集約 | 案件を単位に情報・権限・成果品を管理する |
| 参加者 | 管理者 / 利用者、案件の manager / member / viewer | 案件単位の閲覧・編集・管理権限 |
| 資料 | PDF登録、Markdown資料メモ、プレビュー、削除、RAG化 | 資料を検索可能な案件成果品として扱う |
| CSV | CSV/TSV取込、行閲覧、編集、統合、CSV生成、外部PostgreSQL取込 | 構造化データを案件に紐付け、集計・再利用する |
| チャット | RAG回答、資料抽出、CSV要約/集計/証拠読解、案件横断調査 | 質問に対し根拠と成果品を返す |
| 成果品 | 回答から資料メモ追記、FAQ化、PDF報告書、CSV出力 | 対話を再利用可能な資料・データへ変換する |
| 表現 | Markdown、SSE逐次表示、Chart.js、Mermaid、PDF | 表・図・報告書として伝える |
| 運用 | 入力ガード、品質評価、SQL監査、詳細ログ、ジョブ進捗 | 安全性・観測性・失敗時の説明可能性 |

## 現行固有で移植しないもの

- PHPファイル単位のルーティング、`support.php` の3ペイン画面、PHP session/CSRF実装
- `chat.php` から複数のPHPルートへ分岐する仕組み、現行ログキー、Ollamaモデル名の既定値
- `documents` と CSV/PDF/資料メモを一つのテーブルへ寄せる物理実装
- 現行DBの JSON 行保存、RAGチャンク保存形式、PDF生成のフォールバック連鎖

これらは業務要件ではなく現行の実装選択です。必要な振る舞いだけを新しい境界で再実装します。

## 移行優先度の提案

| 優先度 | 移行する能力 | 理由 |
| --- | --- | --- |
| P0 | 案件・メンバー権限、資料登録/検索、案件チャット、監査 | 業務データの境界と日常利用の基盤 |
| P1 | CSV取込・安全な決定論的集計、資料メモ、報告書生成 | 現行の差別化要素で、業務に直接つながる |
| P2 | FAQ化、CSV統合/AI分類、外部DB取込、図解の高度化 | 利用フロー確定後に追加できる |
| P3 | 複雑な多段推論、LLM judge、全社横断ReAct、watchdog | 効果と運用コストをPoCで評価してから判断する |
